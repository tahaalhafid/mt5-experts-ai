CROSS-REFERENCE RULES
- repository_record_id is the repository primary identifier.
- report_id is the source output/report identifier when one exists.
- One repository record may reference multiple related record ids, but one record must still have one primary repository_record_id.
- Incident-linked records should cross-reference incident_window_reference whenever applicable.
- Proposal-linked records should cross-reference proposal_id_reference whenever applicable.
- Advisory-linked records should cross-reference advisory_packet_id_reference whenever applicable.
- Decision/trade linked records should use decision_id_reference and/or position_id_reference when applicable.
- A record that supersedes an earlier current-summary record should set supersedes_record_id and move the earlier record into historical/superseded role.
- Link-back to source report identifiers is required when a repository object is derived from a prior output contract artifact.

CURRENT VS HISTORICAL INDEXING RULES
- CURRENT_ACTIVE_SUMMARY records must be indexable by current scope and supersedable by newer scope-equivalent records.
- Historical records must retain original generated_at and scope references.
- Superseded records are historical, not deleted by index policy alone.
- Proposal records may transition from current_pending_review to historical_reviewed, but repository_record_id remains stable.
- A record promoted in visibility tier does not become a new truth source; it remains the same repository lineage with updated review status metadata in future implementations.

INDEX PRIORITY INTENT
- VERY_HIGH = incident, proposal, learning evidence, root-cause, anomaly lineages.
- HIGH = current summaries, daily intelligence, drift, three-day reviews, governance audits, council advisory.
- MEDIUM = executive briefs and operator query history.
- Index priority affects future retrieval/index design intent only; no engine is implemented in A5.
