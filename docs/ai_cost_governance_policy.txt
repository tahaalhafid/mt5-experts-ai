RUNTIME BUDGET PRIMITIVES
AIHourlyRunCap=Maximum total AI task runs intended within one hour window across all task families.
AIDeepInvestigationDailyCap=Maximum number of DEEP_INVESTIGATION runs intended within one day window across all task families.
AITriggerCooldownMinutes=Minimum trigger cooldown window before rerunning the same trigger family for the same scope.
max_same_family_runs_per_hour=Per-family hourly run budget used to prevent one task family from starving the others.
max_same_family_deep_runs_per_day=Per-family daily deep-investigation cap.
max_incident_reports_per_window=Maximum incident-style report reruns allowed within one incident window before suppression or downgrade.
max_three_day_reviews_per_window=Maximum institutional review generation frequency within its intended review window.
max_council_advisory_runs_per_bar_or_cycle=Maximum advisory packet generation intent for one candidate/bar/cycle.
budget_pressure_mode=Operational policy mode that controls downgrade aggressiveness and non-critical task deferral.

BUDGET CLASSES
LOW_COST
typical_retrieval_depth=SUMMARY_ONLY
typical_scope_expectation=compact summaries and narrow status surfaces only
cadence_restrictions=may run more frequently within per-family caps
escalation_restrictions=no deep retrieval by default
expected_output_compactness=compact_or_concise
approval_sensitivity=low

MEDIUM_COST
typical_retrieval_depth=SUMMARY_ONLY_or_TARGETED_RETRIEVAL
typical_scope_expectation=summary-first with bounded raw slices only when justified
cadence_restrictions=moderate cooldown discipline required
escalation_restrictions=deep retrieval not default
expected_output_compactness=bounded_medium
approval_sensitivity=medium

HIGH_COST
typical_retrieval_depth=TARGETED_RETRIEVAL_or_DEEP_INVESTIGATION
typical_scope_expectation=multi-slice bounded investigation for high-value review use
cadence_restrictions=rare and cooldown-heavy
escalation_restrictions=deep retrieval only on stronger evidence and within daily caps
expected_output_compactness=bounded_structured_detail
approval_sensitivity=high

CRITICAL_COST
typical_retrieval_depth=reserved_for_rare_critical_incident_investigation
typical_scope_expectation=highest-cost bounded analysis under strongest trigger conditions only
cadence_restrictions=hard-capped and strongly suppressed
escalation_restrictions=never routine, never always-on
expected_output_compactness=incident_focused_structured_detail
approval_sensitivity=very_high

BUDGET PRESSURE MODES
NORMAL
intended_behavior=Use default task budgets, standard cooldowns, and normal escalation discipline.

CONSERVATIVE
intended_behavior=Reduce deep investigation frequency, favor summary-only or targeted retrieval, postpone non-critical cadence work to next window where practical.

RESTRICTED
intended_behavior=Allow only critical event-driven work, compact council advisory, and explicit operator/admin request tasks. Defer non-critical cadence work and proposal work.

DEPTH DOWNGRADE POLICY
- Under CONSERVATIVE pressure, DEEP_INVESTIGATION should downgrade to TARGETED_RETRIEVAL unless the incident is CRITICAL.
- Under RESTRICTED pressure, TARGETED_RETRIEVAL should downgrade to SUMMARY_ONLY where task policy allows.
- Non-critical cadence tasks should defer to the next intended window before rerunning under pressure.
- Proposal generation should defer before consuming deep-investigation budget when advisory budget is tight.
- Council contextual advisory must remain compact and summary-only regardless of pressure mode.
- Output verbosity should shorten under pressure according to A3 compactness rules.

EVENT VS CADENCE PRECEDENCE RULES
- Critical event-driven incident work may bypass normal cadence timing, but must still respect hard daily deep-investigation caps.
- Council contextual advisory may run per candidate event, but only in compact summary-only mode and within per-candidate suppression limits.
- Drift analysis should not rerun full analysis on minor events if the daily cadence window already covers the same scope.
- Governance audit should remain cadence-led and should not be promoted by routine noise already covered by weekly review.
- Operator copilot remains on-request and should not be suppressed by cadence windows when explicitly requested.
- Learning-governance and proposal tasks should prefer cadence/review windows unless a stronger explicit review trigger exists.
