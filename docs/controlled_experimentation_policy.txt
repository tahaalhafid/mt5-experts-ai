1. FORMAL ROLE OF THIS LAYER
This layer defines how accepted learning proposals may become controlled experiment designs only.
This layer does not launch experiments.
This layer does not mutate runtime behavior.
This layer does not promote any learning output into live strategy, governor, failure detector, execution-quality, dirty-environment, trend-continuation, or any other live trading logic.

3. EXPERIMENT ELIGIBILITY RULES
An experiment-eligible candidate requires all of the following:
- learning proposal received ACCEPT_FOR_EXPERIMENT in L4
- proposal remains bounded, traceable, and within declared family/risk framing
- linked evidence bundle is SUFFICIENT_EVIDENCE or STRONG_CORROBORATED_EVIDENCE for the claim being explored
- evidence quality is adequate for the proposed experiment scope
- validation intent exists
- rollback note exists
- no prohibited mutation path is present
- no authority elevation implication exists
- no hidden live rollout intent exists
- limitations and unresolved uncertainty are disclosed honestly

4. NON-ELIGIBILITY RULES
A proposal is not experiment-eligible when any of the following apply:
- proposal was REJECTED
- proposal is DEFERRED without explicit re-entry
- proposal lacks ACCEPT_FOR_EXPERIMENT outcome
- missing rollback readiness note
- missing validation design
- unsupported broad-scope or global optimization claim
- hidden live mutation intent
- unresolved ambiguity too high for the experiment class requested
- evidence is THIN_EVIDENCE or traceability is incomplete
- proposal attempts implicit authority expansion

5. BOUNDEDNESS RULES FOR EXPERIMENT DESIGNS
Every experiment design must declare:
- exact experiment scope
- target subsystem or policy domain
- intended observation window
- what is being evaluated
- what is not being changed
- expected upside
- expected downside
- exit criteria
- rollback readiness note
- review scope
- mutation prohibition note

Boundedness explicitly forbids:
- global "test optimization" requests
- multi-layer undefined experimentation
- hidden live rollout
- open-ended runtime trials
- unspecific change envelopes

6. VALIDATION EXPECTATIONS
Each experiment design must state:
- what success looks like
- what failure looks like
- what uncertainty remains
- what evidence will be collected
- which metrics or qualitative checks matter
- which review window evaluates the result
- which thin-evidence risk remains
Validation remains a governance design requirement only, not a runtime scoring engine.

7. ROLLBACK READINESS REQUIREMENTS
Each experiment design must state:
- rollback_trigger_conditions
- rollback_scope
- rollback_owner_role
- rollback_feasibility_note
- rollback_traceability_requirement
Rollback readiness is mandatory even though this phase does not execute rollback.

8. RELATIONSHIP TO OTHER PHASES
Inputs arrive from L4 ACCEPT_FOR_EXPERIMENT outcomes only.
This layer may prepare bounded experiment designs only.
This layer does not launch experiments.
This layer does not promote changes into runtime.
This layer feeds later promotion or rollback governance in L6 and impact audit in L7.

9. AI ROLE WITHIN CONTROLLED EXPERIMENTATION
AI may help summarize rationale, organize bounded experiment fields, and restate validation and rollback expectations.
AI may not authorize experiment launch, convert design into runtime trial, bypass review outcomes, weaken rollback requirements, or escalate design into live mutation.
