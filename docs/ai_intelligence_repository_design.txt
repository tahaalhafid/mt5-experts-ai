STORAGE SCOPE TAXONOMY
- CURRENT_ACTIVE_SUMMARY = current active intelligence surface with short rolling recency.
- INCIDENT_HISTORY = incident-linked forensic and anomaly history.
- DAILY_HISTORY = cadence-driven daily intelligence history.
- REVIEW_HISTORY = broader periodic institutional review history.
- PROPOSAL_HISTORY = bounded proposals pending/reviewed history.
- ADVISORY_HISTORY = council advisory packet history.
- AUDIT_HISTORY = governance and compliance audit history.
- LEARNING_HISTORY = learning-governance evidence and supporting review history.
- OPERATOR_QUERY_HISTORY = operator-requested copilot response history.

CURRENT VS HISTORICAL DISTINCTION
- CURRENT_ACTIVE_SUMMARY records are current-facing, rolling, and supersedable by newer records of the same scope.
- INCIDENT/DAILY/REVIEW/PROPOSAL/ADVISORY/AUDIT/LEARNING/QUERY history records are historical records with explicit time/scope keys.
- PROPOSAL records may be current_pending_review or historical_reviewed; both remain non-authoritative.
- Superseded/stale records remain historical unless later retention policy marks them purge-eligible.
- Repository objects never become runtime-truth or active-plan truth merely by being the latest intelligence artifact.

REPOSITORY RELATIONSHIP RULES
- Repository records store derived intelligence outputs and references, not live runtime control data.
- Repository current summaries are read-oriented views; they do not mutate runtime truth.
- Repository object families should link back to source task family, output class, and source report identifiers where available.
- Duplicate incident outputs should resolve into one indexed incident lineage rather than uncontrolled duplicate records.
- Council advisory records remain compact and candidate-scoped.
- Proposal records remain reviewable only and never become auto-apply surfaces by storage alone.

RELATIONSHIP TO DELIVERY LAYERS
- Future delivery layers must read from repository-managed outputs, not from ad hoc raw AI text.
- Telegram/email/admin delivery is deferred and not implemented in A5.
- Repository-first design is required before any external delivery integration.

RELATIONSHIP TO LEARNING GOVERNANCE
- LEARNING_EVIDENCE_RECORD and REVIEWABLE_PROPOSAL_RECORD are primary repository supports for later L1-L7 learning-governance phases.
- THREE_DAY_REVIEW_RECORD and GOVERNANCE_AUDIT_RECORD support review-board traceability.
- Future post-change impact records may link back to repository proposal and review identifiers; this is not implemented in A5.

ANTI-SPAM AND COMPACTNESS DISCIPLINE
- Repository storage must remain structured, indexed, and bounded.
- Duplicate incident outputs must not create uncontrolled duplicate repository entries.
- Rolling summary scopes must remain capped and supersedable.
- Compactness expectations from A3 and suppression/dedupe expectations from A4 apply to repository object creation intent.
